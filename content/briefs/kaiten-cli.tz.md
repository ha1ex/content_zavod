## Аудитория

- **Основная:** Инженер команды разработки, который работает в терминале и держит задачи в Kaiten
- **Принимает решение:** Тимлид или технический директор — решение о применении принимает сам пользователь, согласование нужно только для выдачи токена доступа
- **Роли:** Разработчик / инженер команды разработки, Тимлид команды разработки, Инженер автоматизации / DevOps, Аналитик, считающий канбан-метрики, Инженер, который подключает ИИ-агентов к Kaiten

**Боли:**
- Инженер работает в терминале и редакторе кода, а карточку открывает в браузере. Kaiten CLI показывает карточку и документ там же, где идёт работа
- Разовые задачи по десяткам карточек приходится делать руками через веб-интерфейс. CLI выполняет их пакетно из скрипта
- Аналитик собирает историю по каждой карточке отдельным запросом. Локальный снимок в SQLite позволяет посчитать метрики один раз собранными данными
- ИИ-агент дёргает API на каждую карточку и упирается в объём ответа. Массовое чтение и сокращение полей отдают то же самое за меньшее число обращений
- Ответ API большой, а нужно два поля. Флаги compact и fields оставляют только нужное
- Из выдачи инструмента трудно собрать конвейер: разный формат успеха и ошибок. Режим --json даёт единую структуру и статистику stats

**Критерии выбора:**
- Установка одной командой, без сборки из исходников
- Понятная выдача, пригодная для скриптов: JSON, коды ошибок, статистика запросов
- Покрытие сущностей Kaiten: карточки, документы, комментарии, время, теги, чек-листы
- Возможность работать локально и не нагружать API повторными чтениями
- Хранение доступа локально, без передачи токена третьей стороне
- Понятный статус поддержки проекта и канал обращения при проблеме

### Главный сценарий: карточки и документы рядом с кодом
**Карточка открывается там же, где идёт работа**

Команда cards get или documents get с флагом --markdown сохраняет карточку или документ файлом .md в рабочую папку. Инженер читает задачу рядом с кодом и не переключается на доску. Ссылки на вложения приводятся к формату API Kaiten. Команда files download забирает сам файл и докачивает прерванную загрузку. Команда files upload кладёт локальный файл обратно в карточку. Существующий файл не перезаписывается без явного флага --overwrite.

### Локальные снимки и метрики
**Соберите данные один раз — считайте метрики сколько нужно**

Команда snapshot build один раз читает выбранное пространство и складывает данные в локальную базу SQLite. Дальше query cards и query metrics работают по снимку и не обращаются к API. Аналитик считает поток задач за квартал в разрезе досок и перестраивает отчёт, не дёргая историю каждой карточки заново. Наборы данных выбираются под задачу: basic — структура и краткая выборка, analytics — плюс активность и перемещения карточек, evidence — плюс подробности и комментарии, full — всё вместе. Снимок лежит в пользовательском каталоге данных и доступен только вам. Удаление снимка не меняет данные в Kaiten.

### Скрипты и ИИ-агенты
**Одни команды для человека, скрипта и ИИ-агента**

Режим --json отдаёт единую структуру ответа и ошибки, а вместе с ней статистику stats — видно, сколько обращений к API ушло на задачу. Команды batch-get и batch-list читают список карточек за один заход. Флаги compact и fields оставляют в ответе только нужные поля. Диагностика уходит в отдельный поток и не мешает разбирать JSON, а подробную трассировку можно сохранить в файл через --trace-file. ИИ-агент разбирается в возможностях сам: search-tools ищет инструмент, describe показывает параметры, examples даёт готовый вызов.

### Статус проекта и поддержка
**Статус проекта**

Kaiten CLI Community Edition — публичный проект сообщества для автоматизации работы с Kaiten. Проект поддерживает Виктор Огнев, предложения и сообщения об ошибках принимаются в репозитории. На проект не распространяется стандартное коммерческое соглашение Kaiten об уровне обслуживания (SLA), если иное не предусмотрено отдельным соглашением. За официальной корпоративной поддержкой, усилением защиты локальной установки, аудитом безопасности и индивидуальными интеграциями обращайтесь в Kaiten.

### Начать за 15 минут
**Начать за 15 минут**

Шаг 1. Установите: uv tool install git+https://github.com/ViktorOgnev/kaiten-cli.git. Подойдут также pipx и обычное виртуальное окружение с pip. Шаг 2. Проверьте установку: kaiten --version. Если команда не найдена, каталог с исполняемым файлом добавляется в PATH командой uv tool update-shell или pipx ensurepath. Шаг 3. Настройте доступ: kaiten profile add main --domain <домен> --token <токен> --set-active. Домен и токен хранятся в профиле локально. Шаг 4. Прочитайте первые данные: kaiten --json spaces list --compact --fields id,title. Дополнительно: kaiten completion install включает автодополнение для Bash и Zsh.

### Возможности
**Что умеет**

«Команды по сущностям Kaiten» — kaiten <область> <действие> для карточек, комментариев, участников, тегов, чек-листов, блокировок и связей. «Документы и каталоги» — чтение дерева документов и работа с каталогами: таблицами клиентов, оборудования, подрядчиков. «Учёт времени» — чтение и запись журналов времени по карточкам. «Экспорт в Markdown» — карточка или документ сохраняются файлом рядом с кодом. «Поиск возможностей» — search-tools, describe, examples и agent-help объясняют команды прямо в терминале. «Псевдонимы MCP» — привычные имена вида kaiten_list_cards работают как обычные команды. «Кэш чтения» — режим --cache-mode auto сам выбирает срок хранения ответов, --cache-mode refresh берёт свежие данные. «Аккуратный клиент» — ограничение частоты запросов, повторные попытки и явные ограничения времени ожидания.

### Вопросы и ответы
**Вопросы и ответы**

«Это официальный инструмент Kaiten?» — Нет. Kaiten CLI Community Edition — публичный проект сообщества, его поддерживает Виктор Огнев. Коммерческое соглашение Kaiten об уровне обслуживания (SLA) на него не распространяется. За официальной поддержкой и индивидуальными интеграциями обращайтесь в Kaiten. «Нужен ли kaiten-mcp?» — Нет. CLI работает самостоятельно: он не перенаправляет команды в MCP и не зависит от него при работе. Доменный слой у них общий, а источник истины для CLI — собственный локальный реестр. «Где хранится токен доступа?» — В профиле на вашей машине. Профилей может быть несколько, активный переключается командой kaiten profile use. «Что лежит в локальном снимке и кто его видит?» — Рабочие данные Kaiten: карточки, комментарии, история перемещений, журналы времени — в зависимости от выбранного набора. Файл лежит в пользовательском каталоге данных операционной системы, доступ к нему есть только у текущего пользователя. Удаление снимка не меняет данные в Kaiten. «Что нужно для работы?» — Python 3.11 или новее. Установка идёт напрямую из репозитория Git через uv tool, pipx или pip. «Как обновиться?» — Тем же способом, каким устанавливали: uv tool upgrade kaiten-cli или pipx upgrade kaiten-cli. Установка из ветки сама не обновляется. Установку можно закрепить на выпуске: git+https://github.com/ViktorOgnev/kaiten-cli.git@v0.1.27. «Нужно ли учить все команды?» — Нет. search-tools находит нужный инструмент по описанию, describe показывает параметры, examples даёт готовый вызов. Для Bash и Zsh включается автодополнение.
