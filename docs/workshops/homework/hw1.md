# Домашка — «2 ТЗ → 2 лендинга → улучшение → деплой»

**Неделя после воркшопа. Принести результат на воркшоп-разбор.**

Главное в этой домашке — не сами лендинги, а **цикл улучшения**: найти, что не
нравится, починить это в правилах конвейера и опубликовать результат в интернет.

---

## Задание

Возьмите **два ТЗ** (свои реальные задачи или из общего списка) и сделайте
**два лендинга**. По каждому пройдите полный цикл:

```
1. Сделал лендинг  →  2. Нашёл, что не нравится  →  3. Предложил решение
→  4. Внёс и перегенерил (стало лучше?)  →  5. Задеплоил на Vercel
            ↑ если не стало — ещё одна итерация
```

### Шаг 1. Сделайте лендинг

> **«Вот мои материалы — собери из них ТЗ, потом лендинг, и открой в браузере»**

### Шаг 2. Найдите проблемные места

Пройдитесь глазами клиента. Что не нравится? Обычно это:
- **картинка не та** (иллюстрация не из вашего продукта);
- **текст не тот** (штампы, англицизмы, не та подача боли);
- **структура не та** (секции не отвечают на боли читателя).

### Шаг 3. Предложите решение

Определите, какой **шаг конвейера** ошибся (Разбор / Макет / Блоки / Тексты /
Проверки) и какое **правило** надо поменять. Если не уверены — спросите агента:

> **«Какой шаг конвейера отвечает за эту проблему и где его правило?»**

### Шаг 4. Внесите правку и перегенерируйте

По универсальному промпту:

> **«Вот скриншот. Проблема: [что не так]. Как должно быть: [как надо].
> Поменяй правило на шаге [название шага]. Перегенерируй и покажи.
> Если ок — сохрани в main.»**

Не стало лучше — ещё одна итерация (это нормально).

### Шаг 5. Задеплойте на Vercel

> **«Задеплой проект на Vercel и дай ссылку»**

Проверьте, что лендинг открывается по публичной ссылке. Потом внесите любую мелкую
правку, сохраните в main — и убедитесь, что сайт обновился сам.

---

## Часть 2. Правила передачи в разработку (вместе с разработчиками)

У разработчиков есть требования, каким должен быть лендинг, чтобы его приняли без
переделок. Сейчас они живут у них в головах. Добудем их и встроим в конвейер:

1. **Сходите к разработчикам** и попросите **Markdown-документ** с техническими
   требованиями к передаче лендинга: что должно быть в пакете, форматы, нейминг,
   ограничения — всё, что для них важно при приёмке.
2. **Отдайте документ агенту:**
   > **«Добавь эти правила в репозиторий как правила передачи в разработку —
   > чтобы по ним проверялся и собирался пакет на этапе передачи»**
3. **Проверьте на своём лендинге:**
   > **«Лендинг готов — упакуй его для команды разработки по правилам передачи
   > из репозитория»**

Эту часть достаточно сделать **одному-двум людям на команду** (договоритесь, кто) —
правила, однажды добавленные в репозиторий, работают для всех.

## Что принести на воркшоп-разбор

- 🔗 **Две публичные ссылки** на задеплоенные лендинги.
- 📝 **Список «что чинил»**: какие проблемы нашли, какие правила поменяли,
  стало ли лучше после итерации, что не получилось.
- 📦 **Правила передачи в разработку** в репозитории + один лендинг, упакованный
  по ним (от ответственных за часть 2).

Разбор будет полностью прикладным — по вашему материалу. Чем больше реальных
граблей принесёте, тем полезнее будет всем.
