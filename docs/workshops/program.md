# Программа обучения команды: «Контент-завод Кайтен»

Гайд ведущего. Текстовая версия программы — источник истины для презентации
[`program.html`](./program.html). Аудитория: нетехническая маркетинг-команда
(писатели, редакторы, маркетологи, дизайнеры).

## Главная идея

Команда **не печатает команды и не пишет код**. Она открывает проект в Claude Code и
**говорит с агентом по-русски обычными словами** — что нужно. Агент делает всю
техническую часть сам. Цель программы — чтобы команда:

1. **делала лендинги** (описывая задачу словами);
2. **чинила «не очень»** — понимала, на каком шаге сборки сбой, и улучшала правило завода;
3. **передавала готовое** разработке и сохраняла изменения в общий проект.

Терминологию («локальный сервер», «генерация», «коммит в main») команда узнаёт по ходу —
из ответов агента. Понимать слова полезно; печатать команды — нет.

## Сначала — три понятийных вещи (без них дальше тяжело)

1. **GitHub = как Word, только с историей версий.** Всегда видно, кто и что менял, всегда
   можно вернуться назад. А ещё внутри лежит программный код — правила и блоки, по которым
   нейронка собирает лендинги. Меняете правила → меняется результат.
2. **Репозиторий = ваш проект-завод.** Правила, блоки и готовые лендинги в одном месте.
   Лежит и на компьютере, и на GitHub — синхронно.
3. **Пайплайн = сборка лендинга по шагам.** У каждого шага свои правила. Если результат
   плохой — понятно, какой шаг чинить.

### Из чего состоит репозиторий (человеческим языком)

- **Брифы** — задачи от маркетинга (вход).
- **Правила текста (редполитика)** — как писать: язык, тон, без штампов. ← команда правит.
- **Блоки и домены** — готовые UI-картинки и где они уместны. ← команда расширяет.
- **Макеты** — шаблоны структуры страниц.
- **Готовые лендинги** — результат.
- **База знаний** — накопленные уроки. ← команда пополняет.

> Не заставляйте читать файлы по путям. Учите спрашивать агента:
> «проведи экскурсию по проекту», «объясни, что на этапе ‘тексты’».

### Пайплайн генерации (5 понятных шагов)

| Шаг | Что делает | Если плохо | Где правило |
|-----|------------|------------|-------------|
| 1. Разбор задачи | понимает продукт, аудиторию, боли | не та аудитория | бриф + знания об аудитории |
| 2. Выбор макета | структура: какие секции и порядок | не те секции | библиотека макетов |
| 3. Блоки и иллюстрации | UI-картинки из вашего домена | чужая иллюстрация | «домен → блоки» |
| 4. Тексты | заголовки, описания, подписи | штампы, англицизмы | редполитика |
| 5. Полировка и проверки | кнопки, SEO, сборка + автопроверки | подсветит «красные блоки» | встроенные гейты |

## Маршрут

```
ДЗ-0 → Воркшоп 1 → ДЗ-1 → Воркшоп 2 → ДЗ-2 → Воркшоп 3 → Выпускная работа
```

Все домашки — в [`homework/`](./homework/), их можно закидывать команде заранее.

### Домашка 0 — «Подготовьте рабочее место»
Аккаунт GitHub → установить Claude Code → скопировать проект (Conductor / Claude Code) →
сказать агенту «запусти проект и покажи лендинги в браузере» → «проведи экскурсию по
проекту». **Сдал =** скриншот лендингов в браузере. Подробно: [`homework/hw0.md`](./homework/hw0.md).

### Воркшоп 1 — «Как всё устроено + первый лендинг» (~2 ч)
Последовательно:
1. Разбираемся, что такое репозиторий и пайплайн (на пальцах).
2. «Запусти проект и покажи лендинги в браузере».
3. «Собери демо-лендинг по этому брифу и открой в браузере».
4. «Объясни, что ты сделал на каждом шаге сборки».

**Итог:** каждый своими словами собрал первый лендинг. Домашка: [`homework/hw1.md`](./homework/hw1.md).

### Воркшоп 2 — «Свой лендинг от и до» (~2.5 ч)
Последовательно:
1. «Вот мои материалы — собери из них ТЗ и покажи на проверку».
2. Правим ТЗ словами («боли клиента — мягче, без обвинений»).
3. «Собери лендинг по этому ТЗ и открой в браузере».
4. Разбираем, где вышло «не очень» (учимся читать «красные блоки»).

**Итог:** свой лендинг собран, понятно, что улучшить. Домашка: [`homework/hw2.md`](./homework/hw2.md).

### Лендинг вышел «не очень» — что делать (ядро программы)
Завод сам подсвечивает брак — это **«красные блоки»**. По ним видно, на каком шаге сбой:
- ⚠ **не та иллюстрация** → шаг «Блоки и иллюстрации»;
- ⚠ **текст со штампом** → шаг «Тексты» (редполитика);
- ⚠ **не та структура** → шаг «Разбор» / «Макет».

Логика: увидели красный блок → поняли шаг → улучшаете правило этого шага.

### Как доработать правило и сохранить (метод)
Всё — фразами в чат:
1. Скриншот результата.
2. Что не так (конкретно).
3. Как должно быть.
4. Какое правило на каком шаге поменять.
5. «Перегенерируй и покажи, как стало».
6. «Если ок — сохрани в репозиторий (запушь в main)».

**Универсальный промпт** (один на все случаи):
> «Вот скриншот. Проблема: [что не так]. Как должно быть: [как надо]. Поменяй правило на
> шаге [название шага], перегенерируй и покажи, как стало. Если ок — сохрани в main.»

### Воркшоп 3 — «Улучшаем завод и передаём готовое» (~2 ч)
Последовательно:
1. Находим системную ошибку (повторяется → дело в правиле).
2. Дорабатываем правило по универсальному промпту.
3. «Сохрани в репозиторий и запушь в main».
4. «Подготовь готовый лендинг к передаче разработчикам».

**Итог:** команда не только делает лендинги, но и улучшает завод.
Выпускная работа: [`homework/hw3.md`](./homework/hw3.md) — полный цикл соло + одна доработка правила.

## Ежедневный поток (что команда говорит агенту)

1. «Собери ТЗ из моих материалов».
2. «Собери лендинг по этому ТЗ».
3. «Открой лендинг в браузере».
4. «Здесь не так — поправь и покажи».
5. «Сохрани в main и подготовь к передаче».

Ни одной команды руками. Только русские фразы.

## Границы

Статьи **не реализованы** — завод делает только лендинги. Сервис для статей —
отдельная будущая задача (нужна доработка кодом). Эта программа — про лендинги.

## Шпаргалка ведущего (НЕ показывать команде)

Что происходит «под капотом», если агент застрял и нужно помочь руками:

| Человеческая фраза | Что реально делает агент |
|--------------------|--------------------------|
| «Запусти проект и покажи лендинги» | `pnpm install` (один раз) + `pnpm dev` → `localhost:3000` |
| «Собери ТЗ из материалов» | скилл `/kaiten-intake` → `/intake/<slug>` |
| «Собери лендинг по ТЗ» | скилл `/kaiten-generate` → авто-роутинг → `/landings/<slug>` |
| «Проверь / покажи, что не так» | скилл `/kaiten-review`, валидаторы (гейты) |
| «Подготовь к передаче» | `handoff <slug> --require-approved`, страница `/approve/<slug>` |
| «Сохрани в main» | `git add/commit/push` |

Полная техкартина: `README.md`, `docs/pipeline.md`.
