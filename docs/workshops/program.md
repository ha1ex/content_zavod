# Программа обучения команды: «Контент-завод Кайтен»

Гайд ведущего. Текстовая версия программы — источник истины для презентации
[`program.html`](./program.html). Аудитория: нетехническая маркетинг-команда
(писатели, редакторы, маркетологи, дизайнеры).

## Цель программы

Показать команде, что они **сами могут разрабатывать программные продукты**: создавать
себе приложения и инструменты, оптимизировать свою работу — и публиковать это в
интернете. Лендинги — первый практический полигон, но навык шире.

Три навыка, которые даёт программа:
1. **Нейронки для кодинга** — ставить задачи агенту словами и получать рабочий результат.
2. **GitHub** — вносить изменения, понимать версии, не бояться сломать.
3. **Деплой на Vercel** — выкладывать продукт в продакшен (в интернет, по ссылке).

Сквозная цепочка, которую должен освоить каждый:

```
Идея → Нейронка пишет код → GitHub хранит версии → Vercel публикует → Работающий продукт
```

**Итог:** каждый делает себе небольшие программные продукты, помогающие в рабочих
задачах; умеет деплоить в продакшен, вносить изменения, понимает базовые принципы
работы с нейронками, кодингом, GitHub и деплоем.

## Главное правило работы

Команда **не печатает команды и не пишет код**. Открывает проект в Claude Code и
говорит с агентом по-русски: «покажи лендинги в браузере», «поменяй заголовок и
сохрани в main», «задеплой на Vercel». Терминологию узнают по ходу — из ответов агента.

## Понятийная база (до практики)

1. **GitHub = Word с историей версий.** Видно, кто что менял; всегда можно откатить.
   Внутри лежит код — правила, по которым нейронка собирает результат.
2. **Почему одна ветка main:** ветки нужны большим командам разработчиков, чтобы не
   мешать друг другу. Нам важнее скорость и простота — одна общая версия, история страхует.
3. **Репозиторий = последовательный конвейер.** Запрос на входе — лендинг на выходе,
   между ними шаги строго по порядку, у каждого шага свои правила (лежат в репозитории):

```
Запрос (бриф) → 1 Разбор задачи → 2 Макет → 3 Блоки → 4 Тексты → 5 Проверки → Лендинг
```

4. **Плохой результат — диагноз, а не приговор.** Каждый вид брака указывает на шаг:
   - не та иллюстрация → шаг 3 «Блоки» → правило «домен → блоки»;
   - текст со штампом → шаг 4 «Тексты» → редполитика;
   - не та структура → шаги 1–2 «Разбор / Макет» → бриф или макеты.
   Чиним правило шага (не лендинг руками) → все следующие лендинги собираются правильно.

## Карта программы — три стадии

```
Стадия 1 · Инструменты  →  Стадия 2 · Конвейер  →  Стадия 3 · Продакшен  →  Самостоятельность
(В1 + неделя)              (В2 + неделя)            (В3)
```

- **Стадия 1:** пользуемся нейронкой для кодинга; подключили GitHub; вносим изменения
  и проверяем результат.
- **Стадия 2:** делаем лендинги потоком; находим брак и чиним правила шагов; у каждого
  этапа конвейера — свой ответственный.
- **Стадия 3:** деплоим на Vercel; правка → прод обновляется сам; путь к своим мини-продуктам.
- **Итог:** свои небольшие продукты для рабочих задач + понимание базовых принципов.

## Воркшоп 1 — «Основы: нейронка + GitHub» (~2 ч)

Создание лендингов здесь НЕ учим — это домашка. Учим базовую грамотность:

1. **Подключаем GitHub и копируем проект себе** (аккаунт → доступ → клонирование через
   Conductor / Claude Code). У каждого на экране работающий проект.
2. **Вносим первое изменение и сохраняем**: «Поменяй заголовок на этом лендинге и сохрани в main».
3. **Ветки:** что это и почему нам хватает main. Как откатить изменение.
4. **Проверка результата после изменения**: «покажи в браузере, как теперь выглядит» +
   история изменений на GitHub.
5. **Демо ведущего:** как агент собирает лендинг целиком (чтобы домашка была понятна).

**Итог:** каждый умеет менять проект и проверять результат.

### Домашка после В1 (неделя): 2 ТЗ → 2 лендинга → улучшение конвейера

Каждый берёт **два ТЗ** и делает **два лендинга**. Главное — цикл улучшения:

```
Сделал лендинг → Нашёл, что не нравится → Предложил решение (какое правило какого шага)
→ Внёс в репозиторий → Перегенерил → стало лучше? (если нет — ещё итерация)
```

Принести на В2: два лендинга + список «что чинил и что поменял в правилах».
Подробно: [`homework/hw1.md`](./homework/hw1.md).

## Воркшоп 2 — «Разбор полётов + ответственные за этапы» (~2.5 ч)

1. **Смотрим лендинги недели** — каждый показывает свои два.
2. **Разбираем ошибки и находки** — типовые проблемы → к какому шагу конвейера относятся.
3. **Разбираем апдейты репозитория** — кто что менял в правилах, что сработало, что
   конфликтует. Учимся не превращать общий проект в хаос.
4. **Назначаем ответственных за этапы конвейера** — Разбор / Макеты / Блоки / Тексты /
   Проверки: у каждого этапа свой владелец правил.
5. **Каждый изучает состав репозитория** — «проведи экскурсию по моему этапу: где лежат
   его правила и как работают».

**Итог:** поток лендингов без хаоса — у каждого правила есть хозяин.
Домашка: [`homework/hw2.md`](./homework/hw2.md) (углубиться в свой этап + аккаунт Vercel).

## Воркшоп 3 — «Продакшен: публикуем на Vercel» (~2 ч)

Схема деплоя:

```
Проект на GitHub → Подключили Vercel (один раз) → Деплой (ссылка в интернете)
→ Правка → сохранили в main → сайт обновился сам
```

1. **Публикуем лендинг недели в продакшен**: «задеплой этот проект на Vercel и дай ссылку».
2. **Правка → автообновление**: меняем текст → сохраняем в main → прод обновился.
   Так работает связка GitHub + Vercel.
3. **Свои мини-продукты**: у каждого — идея инструмента для своей работы; весь путь
   уже знаком (идея → агент → GitHub → Vercel).

**Итог:** продукт живёт в интернете и обновляется сам.
Выпускная: [`homework/hw3.md`](./homework/hw3.md) — свой мини-продукт на Vercel.

## Универсальный промпт для правок (шпаргалка)

> «Вот скриншот [картинка]. Проблема: [что не так]. Как должно быть: [как надо].
> Поменяй правило на шаге [название шага], чтобы такого больше не было.
> Перегенерируй и покажи, как стало. Если ок — сохрани в репозиторий (запушь в main).»

## Ежедневный поток

```
«Собери ТЗ из моих материалов» → «Собери лендинг по ТЗ» → «Открой в браузере»
→ «Здесь не так — поправь и покажи» → «Сохрани в main и задеплой»
```

## Границы

Статьи **не реализованы** — конвейер делает только лендинги. Сервис для статей —
отдельная будущая задача (нужна доработка кодом).

## Шпаргалка ведущего (НЕ показывать команде)

Что происходит «под капотом», если агент застрял и нужно помочь руками:

| Человеческая фраза | Что реально делает агент |
|--------------------|--------------------------|
| «Запусти проект и покажи лендинги» | `pnpm install` (один раз) + `pnpm dev` → `localhost:3000` |
| «Собери ТЗ из материалов» | скилл `/kaiten-intake` → `/intake/<slug>` |
| «Собери лендинг по ТЗ» | скилл `/kaiten-generate` → авто-роутинг → `/landings/<slug>` |
| «Проверь / что не так» | скилл `/kaiten-review`, валидаторы (гейты) |
| «Подготовь к передаче» | `handoff <slug> --require-approved`, `/approve/<slug>` |
| «Сохрани в main» | `git add/commit/push` |
| «Задеплой на Vercel» | `vercel` CLI / подключение репозитория к Vercel-проекту |

Этапы конвейера (для назначения ответственных) соответствуют фазам пайплайна
P0–P8 (`docs/pipeline.md`): Разбор ≈ P0–P1, Макет ≈ P2, Блоки ≈ P3+P5+P8,
Тексты ≈ P6, Проверки ≈ P7 + валидаторы.
